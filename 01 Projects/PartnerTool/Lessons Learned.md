# Lessons Learned

The expensive ones, so we never pay twice. (NTFS-specific lessons live in [[MFT Fast Scan]]; WPF ones in [[WPF Single-File Quirks]].)

- **Verify fixes against Event Viewer over days, not one check.** The shutdown-crash "fix" at 0.17.30 looked verified; the event log later showed it firing on every close of every later build.
- **WPF binds properties, not fields.** Silent failure — blank UI, no exception.
- **SMBv1 "absent registry value" ≠ enabled.** Modern Windows uninstalls the feature and leaves no `SMB1` value; only flag when the server has it explicitly on or `mrxsmb10` is installed and not disabled. (Scorecard false positive.)
- **OneDrive/cloud files lie about size.** Both `FileInfo.Length` *and* MFT AllocatedSize report full logical size for placeholders — use size-on-disk APIs + the cloud reparse tag.
- **Windows Location gates SSID reads on Win11**, and only the Settings UI can grant desktop-app consent — registry writes don't take. Don't fight it; design around it.
- **Localized display names break matching.** Office language ARP entries ("…para Pequenas e Médias Empresas") must be matched by `culture=` token, not name.
- **`C:\Windows\Installer` is not junk.** Deleting referenced packages breaks repair/uninstall/patching forever — only the Windows Installer COM keep-set method is safe.
- **`Process.WaitForExit[Async]()` waits for the PIPES, not just the process.** With stdout/stderr redirected it also blocks until they hit EOF — and `dism.exe` leaves `DismHost.exe` holding the inherited handles after DISM itself exits. Result (0.19.7): Advanced Cleanup froze at "Running… 100%" with the log already saying *"operation completed successfully"*. Wait on the `Exited` event, then drain the readers with a **bounded** delay. Applies to any tool that spawns a helper child.
- **Timing out with `Task.WhenAny(op, Task.Delay(n))` leaks unobserved exceptions.** The abandoned `op` still faults later (TCP connect → `SocketException 995` on dispose; reverse DNS → "No such host is known" for every IP without a PTR), and the finalizer routes it to the unobserved-exception handler. One subnet sweep wrote **5.1 MB** into `PartnerTool_errors.log`. **Cancel** the operation with a `CancellationTokenSource` instead of walking away from it.
- **For MSIX apps, `File.Exists` ≠ "will run".** winget's binary under `C:\Program Files\WindowsApps\Microsoft.DesktopAppInstaller_…` only grants **execute** to accounts the package is **registered** for. On an RMM/LAPS admin (`~0000AEAdmin`) that never had Store apps provisioned, the file is there but `CreateProcess` returns **"Access is denied"** — and there's no per-user alias and no PATH entry either. Probe by actually running `--version`; if nothing runs, `Add-AppxPackage -RegisterByFamilyName -MainPackage Microsoft.DesktopAppInstaller_8wekyb3d8bbwe` makes the existing package usable for that account (installs nothing).
- **Per-user (HKCU) uninstall strings are an LPE vector** for an elevated tool — refuse to run them.
- **Not every system tool is in System32** — `winmgmt.exe` lives in `System32\wbem`. A resolver that only checks System32 silently falls back to the bare name, which an elevated `CreateProcess` looks for in the app folder and CWD first: the exact hijack the resolver is there to stop.
- **wmic.exe is on death row** — never introduce it; `System.Management` everywhere.
- **Elevated + bare exe names = binary planting risk** — resolve to System32 full paths.
- **Single log at page bottom is bad UX** — per-section inline logs that appear on use won out.
- **Ask-before-delete scope reviews work.** The temp-cleaner folder list was user-approved up front; the one real complaint later (lost crash dumps on a server) was within the approved scope and traceable via the activity log.
- **ShellExecute searches the working directory** — launching a bare name (`taskmgr.exe`, `devmgmt.msc`, `rundll32.exe`) with `UseShellExecute=true` from an elevated app can run a file planted next to wherever the exe was started (Downloads!). Found in the 0.17.54 full review across 4 launch sites. Always pin via `ProcessRunner.ResolveSystemTool` (any extension; System32 → Windows-dir fallback for regedit).
- **PS 5.1 `Get-Content`/`Set-Content` mangle UTF-8.** They default to the ANSI code page, so reading a UTF-8 file (·, —, ▶, ●) and re-writing it double-encodes every non-ASCII char into mojibake (`Â·`, `â€"`). This silently corrupted the vault code snapshot and two notes — and the round-trip test *passed* because it used the same lossy cmdlets on both sides (self-consistent garbage). Always use `[IO.File]::ReadAllText` / `[IO.File]::WriteAllText` for UTF-8, and make the verification read the same way or it hides the bug. Reverse existing mojibake with `[Text.Encoding]::UTF8.GetString([Text.Encoding]::GetEncoding(1252).GetBytes($text))`.
- **PS 5.1 array-literal precedence:** in `@("a", $x + $y, "b")` the comma binds tighter than `+`, splitting `$x + $y` into two elements. Silently corrupted the vault code snapshot's fence lines — precompute the concatenation. Caught only because the restore was round-trip **tested**.
- **Test disaster recovery, don't assume it.** The first snapshot looked fine when browsed but couldn't restore cleanly; the round-trip diff found it in minutes.
- **Don't copy "PC optimizer" features wholesale.** Researched IObit Advanced SystemCare for the Health Check tab: registry clean/defrag is Microsoft-unsupported "snake oil" (Malwarebytes' words), privacy/browser wiping is an MSP liability, "internet boost/system optimization" is placebo. Built Health Check as a curated front-end over checks we already trust (junk/disk/updates/security/stability) + one new safe capability (broken-shortcut finder, reversible recycle) — no registry/privacy/optimizer theatre.
- **Dell's "unaccounted GBs" are TWO different things (0.19.0).** (1) `C:\ProgramData\Dell\SARemediation\SystemRepair\Snapshots` — SupportAssist OS Recovery's *own* repair points, **not VSS**. Default reserve 15 GB, Dell staff call 20–25 GB "normal", and a known purge bug reaches 85–100 GB+. Dell is explicit: **do not delete the folder by hand — it breaks OS Recovery**; the supported fix is turning **System Repair off in SupportAssist**, which purges it on reboot. So we only report it + link out. (2) **Unbounded VSS shadow storage** (Dell KB 000129138) — VSS ships with `MaxSize` unset, so when SupportAssist OS Recovery reads VSS it goes *unbounded* and fills the drive as "unaccounted" space. Dell's own fix — `vssadmin resize shadowstorage /For=C: /On=C: /MaxSize=<pct>%` — is safe to automate, so that's the one part we offer as a (gated) fix. Dell's KB says **2%**; per user we default to **5%** (`DellRemediation.VssMaxPercent`) for more restore-point headroom. Query the limit via `Win32_ShadowStorage` matched to `Win32_Volume` DeviceID, never wmic.
- **Two Windows-servicing operations at once = one fails.** A tech ran Full Repair and Update All simultaneously (different pages, separate busy-locks) and DISM RestoreHealth died with **0x800F0915** while PSWindowsUpdate was installing — CBS/TrustedInstaller can't serve both. Per-page busy state isn't enough; anything driving the servicing stack needs an app-wide gate (`ServicingLock`, 0.17.68).
- **The Reliability index isn't "are you crashing?"** `Win32_ReliabilityStabilityMetrics` drops for every app install, update and driver change — a machine that installs software all day (like this dev box) reads ~1.5/10 while never crashing. Verified via `Get-CimInstance` (newest 1.685, 7-day peak 9.92). Don't "fix" the number — add context (7-day best + a caption) and point at actual crash history.
- **Finding winget.exe when elevated is a trap (0.19.5).** The common `%LOCALAPPDATA%\Microsoft\WindowsApps\winget.exe` is a *per-user* App-Execution-Alias — absent when the tool runs elevated as an admin account that never launched winget, and a bare `winget.exe` isn't on the elevated PATH either. Field failure on a Win10 box: "winget.exe … The system cannot find the file specified" (software export + Update All). The machine-wide binary is under `C:\Program Files\WindowsApps\Microsoft.DesktopAppInstaller_…`, **but that folder DENIES directory listing even to Administrators** (verified: `GetDirectories`/`icacls` both throw AccessDenied without taking ownership) — while still allowing **traverse/execute**. So you can't enumerate it, but you *can* run a known full path. Get the exact path from `Get-AppxPackage -AllUsers Microsoft.DesktopAppInstaller | … .InstallLocation` (a string, no listing) then `File.Exists(dir\winget.exe)`. Consolidated into one `WingetLocator` (was 3 different resolvers, 2 of them weak). Lesson: had THREE copies of this resolution logic and only the vendor-update one was robust — single source of truth matters.
- **A single-file exe cleaning temp will hit its own DLLs.** PublishSingleFile self-extracts native libs to `%TEMP%\.net\PartnerTool-<ver>\<hash>\` and holds them open, so a temp cleaner "skips (in use)" its own `wpfgfx_cor3.dll` etc. — harmless but confusing in the log. Skip `AppContext.BaseDirectory`'s extraction folder (`TempCleaner.IsSelfExtract`). Only matters in the shipped single-file build; `BaseDirectory` isn't under `\.net\` in a dev build.
- **`Remove-Item -LiteralPath '…\*'` deletes nothing.** `-LiteralPath` takes the path *literally* — the `*` is not a wildcard, so it looks for a file literally named `*`, finds none, and (with `$ErrorActionPreference='SilentlyContinue'`) silently no-ops while any following "cleared" message lies. To empty a folder's contents use `-Path '…\*'` (globs) or `Get-ChildItem -LiteralPath dir | Remove-Item`. Bit the Feature-Update Leftovers DO-cache fallback (0.19.2). LiteralPath is right for the *targets* (exact paths, no glob); wrong the moment you append `\*`.
- **Health checks must not cry wolf.** The Dell VSS check flagged shadow storage as "UNBOUNDED" whenever there was simply *no* shadow-storage association — which is the normal state on any machine with System Protection off. That's a false alarm that deducts health points and offers a pointless "fix" on healthy PCs, which erodes trust in the tool in front of a client. Fix (0.19.2): distinguish *not configured* (report as fine) from *association exists and is uncapped* (the real Dell KB 000129138 problem) — only the latter is flagged/fixable. General rule: when a signal has a benign interpretation and an alarming one, prove the alarming one before firing.
- **`CreateEntryFromFile` opens with only `FileShare.Read`** — zipping a file another handle has open for append (our own activity/error logs) throws a sharing violation. Read with `FileShare.ReadWrite` and copy the stream instead, or the tool silently drops its own live logs from the diagnostic bundle.
- **"Microsoft" manufacturer ≠ Surface.** A Windows 365 Cloud PC / Hyper-V / Azure VM reports `Win32_ComputerSystem.Manufacturer = "Microsoft Corporation"`, which matched the Surface firmware tool. Match the OEM tool on **manufacturer + model**, detect VMs by model/maker signature ("Virtual Machine", "Hyper-V", "VMware", "VirtualBox", "Cloud PC"…) and offer no tool for them, and only pick Surface when the model literally contains "Surface".
- **Listing Store updates without installing them:** `AppInstallManager.SearchForAllUpdatesAsync()` (parameterless) *starts installing* what it finds. For a passive scan use the 3-arg overload with `AppUpdateOptions.AutomaticallyDownloadAndInstallUpdateIfFound = false` — it returns the pending UWP/MSIX updates read-only, without kicking off downloads or cancelling ones Windows already started. **But a fresh search misses what the Store already queued itself (0.19.11):** updates sitting in the Store's own install queue — stuck at ReadyToDownload, or errored with 0x80073D02 ERROR_PACKAGES_IN_USE ("App was in use") — come back **only** from `AppInstallManager.AppInstallItems`, never from a new search. Proven with a probe on the dev box: search = 0, queue = 3, exactly the 3 the Store UI showed. Read both and merge; for the installer path, `AppInstallItem.Restart()` retries a stuck item.
- **"Windows cannot access the specified device, path, or file" on a fresh build = ASR, not corruption.** Check the Defender Operational log for event **1121**: the prevalence/age/trusted-list ASR rule (`01443614-…`) blocks new unsigned exes — every PartnerTool build is a brand-new hash. Diagnose before rebuilding; fix via Intune per-rule exclusion or code signing (see [[Build & Release]]).
- **Timestamped log filenames need seconds.** Minute-granular names meant a temp Scan followed by Clean in the same minute overwrote the scan's log — silent evidence loss that only shows up when someone needs the earlier file.
- **Changelog dates rot when copy-pasted.** 26 `versions.md` entries all said "2026-06-08"; nobody noticed until a review cross-checked them against Event Viewer. When a ritual includes a date, verify it's *today's*.
- **DispatcherTimer gotchas:** parameterless `DispatcherTimer()` is **Background** priority; and a timer started only from `IsVisibleChanged` is fragile. For must-run timers use `DispatcherPriority.Normal`, start in the constructor, and gate the *work* on visibility. (System Info's "updates every minute" silently never fired.)
- **`IsEnabled` inherits — a Stop button inside the panel you disable is dead on arrival.** The traceroute Stop button lived inside `PnlNetDiag`, which the click handler disables for the whole run; WPF child controls cannot re-enable themselves under a disabled parent, so the one control meant to interrupt the run was disabled WITH it (UIA threw ElementNotEnabledException — caught by the scripted UI test before any human hit it, 0.19.13). Scope the disable to the inputs and keep interrupt/output controls outside it.
- **A plain WPF `Border` never enters the UIA tree — `AutomationProperties.AutomationId` on it is a no-op.** Setting an AutomationId/Name on a clickable Border (a live-stat tile) did nothing: it appeared in neither the control nor the raw view (verified by dumping all 219 elements — the Border wasn't among them, while Buttons and Text elements were). WPF only gives peers to control-type elements. For a clickable tile to be testable AND keyboard/screen-reader accessible, make it a real `Button` (templated to look identical) — that gives control-view presence, focus, Invoke, and an announced Name. Discovered via the scripted UI driver (0.19.14).
- **Narrow LibreHardwareMonitor to the subsystems you actually read.** We enabled `IsMotherboardEnabled` + `IsControllerEnabled` but only ever read CPU/GPU/Storage nodes. That probing makes the library query `Microsoft_IPMI` (a BMC/server-only WMI class) on every collect, which fails `invalid class` on client laptops and floods `Microsoft-Windows-WMI-Activity/Operational` — traced to our own PID. A two-config probe proved the motherboard/controller flags yielded ZERO extra temperature data on the test machine (CPU/GPU temp were NONE either way; only NVMe storage temps exist). Enable only what you consume: less WMI noise, faster init, smaller native-driver surface (0.19.14).
- **A fresh dev box has no `nuget.org` source → `dotnet build/publish` dies with `NU1101`** (can't find LibreHardwareMonitorLib / the runtime packs), because a bare machine's NuGet config lists only offline/VS feeds. Don't hand-add the source each time — **pin it in-repo with `nuget.config`** (`<clear/>` + nuget.org) so every clone restores hermetically. Bit us standing up the [[Claude Dev VM]] (2026-07-18).
- **A fresh clone needs per-clone git config the repo can't carry:** `user.email`/`user.name` (else the first commit fails *"Author identity unknown"*) and `core.hooksPath .githooks` (else the vault auto-sync hook never runs). Both are in [[New Machine Bootstrap]] now.
- **`.bat` build scripts ending in `pause` hang any non-interactive run** — the pause waits for a keypress that never comes. `publish.bat`/`publish-singlefile.bat` are for humans double-clicking; for scripted/agent builds call the underlying `dotnet publish` line directly.
